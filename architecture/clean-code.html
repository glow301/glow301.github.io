    谈代码整洁之道

    2020/9/13 读书笔记架构

    在我们写代码的时候，代码的可读性，维护性似乎是永远避不开的话题。尽管我们每个人都想写出易于维护，可读性高的代码，但是在实际项目中，结果往往都不尽如人意。每当我们接手一个别人的项目时，几乎都是充斥着批判，心里说着是谁写的这么烂的代码。但是轮到我们自己的时候，可能结果真的比前人好不了多少，等到后人接手我们的项目时，十有八九也会像我们当时接手一样，心里默默的谴责着。

    如果要用一句话总结这本书，我觉得应该就是代码是写给人看的，只是恰好可以被机器执行而已。任何一个人都可以写出计算机可以理解的代码，我们要写出人类容易理解的代码。

    书中给我印象比较深的一个例子，在我们日常工作中，其实真正写代码所花费的时间是很少的，大部分时间其实是在滚动屏幕，看其他地方的代码。也就是说我们读代码的时间要远多于写代码的时间，可见代码的可读性是多么的重要。

    下面是给我印象比较深和我比较认可的一些点，可能不是完全都出自代码整洁之道这本书。

    # 代码的上下文

    假设我们现在有一个求学生考试平均分的函数。

    func getStudentsAverageScore(stuIDs []string) (float64) {
    var totalScore, averageScore float64

    if len(stuIDs) == 0 {
        return 0
    }

    for _, s := range stuIDs {
        score := getStudentScore(s)
        totalScore += score
    }

    averageScore = totalScore / len(stuIDs)

    return averageScore
}

    • for 循环里的变量 s，虽然变量名看起来没有什么实际含义，但是定义的地方和使用的地方只差了 1 行，而且逻辑很简单，所以在阅读的时候并没有感到障碍。但是如果这个循环中又嵌套了一个循环，然后变量 s 在 50 行之后才会使用到，那么阅读起来就会有一些费劲了，你需要不断滚动屏幕来查看变量的声明。
    • 同理，变量 averageScore 距离声明处只有 10 行左右，所以不会感到有障碍。但是如果变量的使用和声明中间差了上百行，那么读起来就相当费劲了，不仅需要来回滚动屏幕，有时候甚至你不知道它是一个局部变量还是全局变量。

    所以，尽量让变量在使用的时候再声明，这样可以避免一些上下文的切换，让别人更好的理解。

    # 命名

    命名其实真不是一件容易的事，有时候为了相出一个好名字，需要花费不少时间。总结一下，有这么几个方面:

    # 命名要有意义

    对比下面两个例子

    var d time.Time // the expire date
var expireDate time.Time
    1
    2

    明显第二张情况要好于第一种，对于不好的命名，我们往往需要通过注释来解释它。

    # 避免变量名中带有类型

    看下面这段代码，哪一种更好呢？

    var accountList []int
var accounts []int
    1
    2

    其实，[]int 类型已经告诉我们，这个变量是一个 list 了，没有必要在变量名中说明类型。 即使使用的是弱类型语言，通过名词的复数形式，也可以说明是一个 list。如果 accounts 不能描述清楚的话，用 accountList 一定会比 accounts 好吗？

    # 上下文不要太远

    上面已经说过，变量的声明和使用距离太远会增加我们阅读的成本。尽量让变量的使用和声明可以在一屏内找到。

    # 函数

    只要写代码，你一定会用到函数。我们要做的，就是要保持函数尽可能的简单。这里说的简单，不仅仅是逻辑简单，还包括函数的长度，参数的个数等，相信没有人会愿意去读一个上千行的函数吧。

    # 单一职责

    一个函数就应该做好一件事，这样才能保持简单。一个很典型的反例是通过标识符来判断函数的行为。

    看下面这段代码

    func foo(xxx int, flag bool) {
    if flag {
        // do something 
    } else {
        // do something
    }
}

    另一个例子是查询和修改，即在一个函数中同时做了查询和修改的操作，比如这样。

    func getAndUpdate(query string) (string, error) {
    // ...

    比较推荐的做法是将查询和修改分离。我们可以在一个函数中执行查询和修改，但是应该是调用两个独立的函数。可以记住这样一句话：任何有返回值的函数，都不应该有看得见的副作用。

    # 参数个数

    函数参数的个数应该尽可能的少。1 个参数要比两个参数的简单，如果函数的参数超过 3 个，就需要考虑把其中一些参数封装成类了。

    多个参数的函数，特别是在这些参数的类型相同的时候，会更加难以区分。

    func foo(a, b, c, d, e, f int) {
    // ...
}

    在调用的时候，会有很大概率将参数的顺序搞错。相信有很多人在使用 assert.Equal(expect, actual) 的时候，经常搞错 expect 和 actual 的顺序。

    # 注释

    什么也没有良好的注释有用，什么也没有糟糕的注释更有破坏性。

    在代码迭代的过程中，注释很难保证得到及时的更新。而且，有时候注释并不准确，甚至是在说明错误的事。真正可以相信的只有代码，因为代码才是真正执行的。所以，当需要写注释的时候，可以先想想能不能用代码表达。

    我曾经在国内某 top2 大厂见过如下代码

    # 应该有的注释

    • 法律/版权信息
    • 对意图的解释
    • 警示
    • TODO (需要及时清理)

    # 不应该有的注释

    • 多余的注释（不能提供比代码更多的信息）
    • 误导性注释（更新不及时，注释不精确）
    • 日志式/HTML 注释
    • 注释掉的代码（已经有版本控制系统了）

    # 格式

    代码的格式，不仅仅包括代码的风格，还应该包含代码文件的长度，组织方式等。

    • 相关的概念应该放在一起，而不是分散在各个文件中。
    • 相关的函数应该放在一起，调用者应该放在被调用者上面。
    • 使用空格/空行来增加代码可读性。

    # 单元测试

    测试代码和生产代码一样重要。它一样需要被思考，被设计和照料。应该像生产代码一样保持整洁。

    # 每个测试一个断言

    单个测试中的断言数量应该最小化。尽量保持在一个测试函数中只测试一个概念，避免出现超长的测试函数。

    # FIRST 原则

    • 快速（Fast）
      • 测试应该可以快速运行。测试运行缓慢，你就不会想要频繁运行它。
    • 独立（Independent）
      • 测试应该互相独立。某个测试不应该为下一个测试设定条件。
    • 可重复（Repeatable）
      • 测试应该在任何环境中重复通过。测试应该可以在生产环节，开发环境，无网络的环境通过。
    • 自我验证（Self-Validating）
      • 无论测试通过还是失败，都不应该通过日志来确认测试是否通过。
    • 及时（Timely）
      • 测试应该及时编写。

    # 小结

    软件项目的主要成本在于长期维护，所以保持代码的整洁是非常重要的。

    • 函数的长度，参数个数，类的大小应保持尽可能的短。符合单一职责。
    • 注释不能美化糟糕的代码，如果需要注释，先尝试是否可以用代码说明。
    • 单元测试和生产代码一样重要，也应该保持整洁。
    • 系统应该是由多个短小的类，而不是几个巨大的类组成。

    由于作者水平有限，不能列出书中所有的点。如果只能记住一句话的话，那就是代码是写给人看的，只是恰好可以被机器执行而已。
